## Roadmap Phase 2 (second pass) — Dietary, cooking-for & favourites

All six scoped deliverables for this pass are built, tested, and committed across four runnable checkpoints. CP1–CP3 were application-code only (the dietary_tags, user_dietary_tags and cooking-for columns were already in the baseline, as the auth-pass forward note predicted). CP4 — a per-user favourites correction discovered mid-pass — needed the first migration since the catalogue phase. Deliberately unstyled, two-person trust model, as scoped.

- **Dietary tag CRUD** (`/manage/dietary-tags`) — add / edit / delete, each typed `allergy` or `preference`, alongside the other manage screens. A `TAG_TYPE_CHOICES` constant in models is the single source of truth (mirrors `SECTION_CHOICES`), shared by the form and the brief split. Case-insensitive duplicate guard, consistent with ingredients/categories/users.
- **Per-user tag assignment** — on the existing user-edit screen, checkboxes grouped by type. Editable for active users; retired users show their tags read-only (no form) with the route guarded too.
- **Wizard "cooking for" upgrade** — the plain servings number is replaced by a multi-select of active users + a guest counter. Servings is now *derived* (selected users + guests), not stored. Combined dietary constraints = the union of selected users' tags. Zero users + guests is valid; at least one cover is enforced; total capped at 20. `cooking_for_user_ids` and `guest_count` are now persisted on the Generation (previously hardcoded empty).
- **Dietary into the brief (minimal)** — `combined_dietary()` is the single source for the union, split into hard allergies vs soft preferences. The dynamic prompt gets a forceful absolute-exclusion block for allergies and a gentle steer for preferences; the system prompt gains one supporting rule line. No creative-seed / anti-repetition — that's the Phase 3 deep-dive, left untouched.
- **Favourites** — a toggle on the recipe page and a favourites list. Reworked to per-user during this pass (see below).
- **History filter by cook** — a `?user=` filter on `created_by_user_id`, with a dropdown that includes retired users (they still hold history).

## The favourites deviation (CP4 — out of scope, folded in)

The original brief specified favourites as a single `recipes.is_favourite` boolean (spec §3, §6). Testing surfaced that this is mis-specified: recipes are *shared* objects (two per generation, either viewable by anyone), so a flag on the recipe row is global — one household member's favourite showed up in the other's list. A flag can't record *who* favourited what.

The fix replaced it with a `recipe_favourites(user_id, recipe_id)` association table and a `User.favourite_recipes` relationship (backref `favourited_by`) — the same shape as `user_dietary_tags`. The toggle now adds/removes the *current user*; the list and the star markers (recipe page + History) all reflect the logged-in user. `Recipe.is_favourite` was removed from the model entirely rather than left dead, so a freshly created DB (the Docker / `create_all` path) carries no vestigial column. No backfill — the handful of test-era favourites were discarded by choice.

This is the one strictly out-of-scope item: a correction to the favourites mechanic beyond the original brief. Two smaller out-of-scope touches: the history filter dropdown including retired users, and a latent-bug fix (`default_user_id()` in `service.py` referenced `User` without importing it — fixed when `User` was imported for the dietary union).

The batch-cook option (spec §4) was explicitly left out of this pass, as agreed at the start.

## Things learned along the way (not accounted for in scope)

1. **The favourites mis-spec — the reusable lesson.** Any per-person "flag" on a *shared* object needs a join table, not a column on that object. Same shape as dietary tags. Worth checking future "is_x" booleans against this before adding them to a shared row.
1. **A new association-table migration is the easy kind** — purely additive `create_table`, fully autogeneratable, none of the NOT-NULL-column server_default dance. Autogenerate (with `compare_type`) produced *only* the two intended changes, no phantom diffs on other tables.
1. **…but the autogenerated `downgrade()` hits the phase-2 trap.** Re-adding the dropped `is_favourite` as `NOT NULL` with no default fails on a populated table. The downgrade needs `server_default=sa.false()` added by hand for a rollback to work — the same add-with-default principle, now applicable on the *down* path. (Round-trip down→up was verified with that fix.)
1. **Dropping a column on SQLite works via `render_as_batch`** (table rebuild); data in all other columns survives. Confirms the batch setup handles drops, not just the adds we'd done before.
1. **Two schema-build paths must stay converged.** `create_all` (from the models — the Docker/fresh path) and `flask db upgrade` (migrations — existing DBs) have to produce the same schema. Removing `is_favourite` from the *model* and dropping it in the *migration* keeps a fresh install and an upgraded install identical; doing only one would leave autogenerate reporting a permanent diff. A standing check for every future schema change.
1. **The wizard endpoint rename has three touch-points.** `step_servings` → `step_cooking_for` meant updating the route, the forward redirect (in `step_time`), and the back-link (`review.html`). `url_for` catches a missed reference at render time, which is the safety net.
1. **Old-shape sessions survive a deploy.** Because `SECRET_KEY` is stable, a tablet sitting mid-wizard keeps its pre-upgrade session dict (had `servings`, lacked the cooking-for keys). Handled gracefully — session reads use `.get()` with defaults (so they read as 0 covers), and the new generate-guard bounces a zero-cover brief back to the cooking-for step rather than generating for nobody. General principle: wizard session reads should tolerate shape changes across deploys.
1. **Tag delete semantics differ from category/user delete.** A dietary tag has no immutable historical dependency — generations store user ids, not a tag snapshot — so hard-delete + auto-unassign (SQLAlchemy clears the join rows) is safe. This is deliberately *unlike* categories (NOT NULL FK guard) and users (retire-only for history attribution).

## Forward notes

- **Spec correction needed:** §3 (`recipes.is_favourite`) and §6 (global favourites) are superseded by the per-user `recipe_favourites` join table. Update before the spec is used as a reference again.
- **Dietary brief wiring is deliberately minimal.** Tag *names* are passed verbatim (e.g. "Peanut allergy" stated as a literal exclusion). The Phase 3 prompt deep-dive should refine how tags map to actual ingredient exclusions, alongside creative-seed and anti-repetition.
- **Still open from earlier, untouched here:** server-side idempotency / async generation (Phase 5), local-time display / UTC→BST (Phase 4 — History *and* the new Favourites list still render UTC), and equipment-into-the-brief (Phase 3).
- **No roles/permissions:** any logged-in member can manage tags, assign tags to anyone, and favourite freely — consistent with the two-person trust model, but flag if that ever changes.
- **CP4 is the only checkpoint where deploy order matters:** model edits + migration must both be in place before the app serves (`git pull` → make edits → `flask db migrate` → fix the downgrade line → `flask db upgrade` → restart).
