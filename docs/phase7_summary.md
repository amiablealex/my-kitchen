Roadmap Phase 4, Pass 4b (recipe structure, prompt voice & eval harness) is complete and committed across three checkpoints, building on the 4a brief-assembly work summarised in phase6_summary.md. The production target throughout was Gemini, tuned and compared on a Google AI Studio key.

CP1 (recipe structure) was the deterministic, schema-touching checkpoint. The recipe contract gained two fields: a required-but-leniently-handled `intro` (an energetic context paragraph — the why, the culinary framing, what makes the dish good) shown at the top of the full recipe, and an optional `tips` array of `{title, text}` (serving suggestions, finishing touches, troubleshooting) shown at the bottom only when present. Storage is nullable `recipes.intro` (Text) and `recipes.tips_json` (JSON), added via a small additive Alembic migration — both nullable, so no NOT-NULL backfill dance and old rows simply render without those sections. The migration was autogenerated (so the revision chain picked up the real head automatically) and reviewed down to "only the two nullable adds, no phantom diffs" before applying, keeping create_all and the migration converged. Validation treats `title`/`ingredients`/`cook` as the only hard-fail structural fields; `intro` and `tips` never trigger the retry — a cosmetic omission shouldn't nuke an otherwise-good two-recipe response — though the prompt still asks forcefully for an intro so empty-intro stays the rare exception. The mock provider was updated to emit both new fields (one recipe with tips, one with an empty list) so both display paths are exercised offline.

CP2 (prompt voice + temperature) was the craft. An earlier voice draft was extended for the new intro/tips structure and isolated as a single clearly-bounded `VOICE_AND_WRITING` constant in prompt.py, so CP3 iteration only ever touches one block and the hard 4a rules (the §5.5 precedence ladder, the output contract) stay untouched alongside it. The voice target is the examples.md register: warm and energetic, explains *why* a dish works, references culinary traditions where genuinely apt, gives doneness cues from the senses rather than only timers, and evangelises finishing touches — all of it living *inside* the JSON fields (intro, step text, tips), never as loose prose around the JSON. A config-driven `LLM_TEMPERATURE` (default 0.8 — moderate on purpose; variety comes from the creative seed and anti-repetition per §5.3, not from cranking heat) was wired into the real Anthropic and Gemini providers, and `LLM_MAX_TOKENS` was added (the richer output is longer — guard against truncation). The mock ignores both.

CP3 (eval harness + golden briefs) is the re-runnable `flask eval-recipes` CLI that loops a set of standalone golden-brief dicts straight through the user-prompt builder and provider, bypassing the DB→brief assembly that 4a already tested. Per brief it reports a hard-constraint PASS/FAIL checklist — valid JSON to the schema, not truncated, exactly two recipes, every must-use item present in both, no allergen present, ≤2 to_buy per recipe, two distinct titles — plus the full recipes printed for eyeballing voice quality, and saves a timestamped report so prompt iterations can be diffed. Assertion metadata (required/forbidden keywords) is kept separate from the clean prompt-input dict, because a tag like "Peanut allergy" won't substring-match "peanuts" in the output, so the allergen check needs its own explicit forbidden list. Reproducibility is locked down by pinning a fixed creative seed per brief and fixing recent_titles empty, and by running the eval at a low default temperature (0.2) so the only moving variable between runs is the prompt; final confirmation is then a sanity pass at production temp (0.8). The starter set of eight briefs covers the examples.md cases plus the nasty ones: allergen-as-must-use (peanut + peanut-allergic cook → assert no peanut), empty pantry (staples-only), blank must-use (whole-pantry), preference-override (vegetarian + chicken must-use → chicken used), a large must-use list (distinctness stress), surprise-me cuisine, and batch cook. Each is documented in examples.md with a one-line "what good looks like" note so eyeballing is a checklist, not vibes. examples.md itself was added to the repo this phase (it had only ever been a local reference): it now serves as both the voice north-star and the eyeball checklist.

Issues hit and resolved. The first real Gemini eval failed 0/8 with cryptic "Expecting ',' delimiter" JSON errors. The cause was not malformed JSON but truncation: the richer 4b output ran past the initial `LLM_MAX_TOKENS` of 4000 and got cut off mid-string, leaving syntactically incomplete JSON — the parser was reporting a symptom, not the disease. This also reproduced in the live wizard, confirming it was provider-level, not eval-specific. Fixed in two parts: a `finish_reason` truncation guard so the service notices `max_tokens`/`length` finishes, doesn't waste its one retry on them (retrying won't help — the output is just too long), and surfaces an honest "the recipes came back longer than the limit and got cut off" rather than a parser complaint; and raising `LLM_MAX_TOKENS` to 8000. A related red herring: a `dir()`-based check on Gemini's `GenerateContentConfig` returned blank and suggested `max_output_tokens` wasn't a valid field, but on a Pydantic v2 model the real fields live in `model_fields`, not `dir()` — checking `model_fields` confirmed both `temperature` and `max_output_tokens` were correct all along (google-genai 2.8.0). After the cap fix, all eight briefs passed on both models.

Model decision. The default Gemini model was switched from `gemini-2.5-flash` to `gemini-3.5-flash`, accepting some cost/speed for clearly stronger culinary reasoning and voice — chosen on eval evidence, not vibes. 3.5-flash produced genuinely distinct second recipes (a lentil ragu over sweet-potato mash rather than a near-duplicate of the first), interpreted the creative seed into dish structure rather than name-checking it (building the peanut-brief dish around a heat-cooling cucumber foil), and taught real technique with the why (blooming cracked coriander seeds; velveting chicken with a food-science tip) — the examples.md register. It held up at production temp 0.8, where the extra freedom produced *better* reasoning (an explicit Greek-briam reference, umami soy in a Med bake) rather than incoherence. The change was made at the code level (`DEFAULT_MODELS["gemini"]`) so fresh installs and the eventual Docker deploy get the good model without depending on a `.env` var; `LLM_MODEL` remains available for future A/B runs.

Settings landed: `LLM_PROVIDER=gemini`, model `gemini-3.5-flash` (code default), `LLM_TEMPERATURE=0.8` (production), `LLM_MAX_TOKENS=8000`. No schema change beyond CP1's two nullable columns. Allergy handling remains prompt-level plus the display caveat (deterministic per-ingredient allergen blocking stays a deferred future hardening, §5.5).

Key learnings and principles.
- **Truncation can masquerade as malformed JSON.** A response cut off at the token ceiling leaves incomplete-but-not-malformed JSON, so the parser reports a delimiter error. "Not truncated" must mean "the model said it finished" (finish_reason), never "the JSON happened to parse." The guard is now permanent.
- **Check Pydantic v2 model fields via `model_fields`, not `dir()`.** `dir()` lists methods, not declared fields, and gives a false-negative on field-existence checks.
- **An eval harness turns prompt-tuning from vibes into evidence.** It caught the truncation bug, and it made the model comparison a clean, reproducible side-by-side. Pinning seed + recent_titles + low temperature so the prompt is the only variable is what makes run-to-run diffs meaningful.
- **max_tokens is about output length, not model cost.** A more expensive model doesn't need a higher cap; size the cap to the longest real output (the batch-cook brief is the canary) and leave headroom. 8000 clears every current brief with a natural STOP.

Watch-items for the future (noted, not acted on).
- **`to_buy` lean.** 3.5-flash reaches outside the pantry for "nice" extras a little readily (feta, cucumber, even cornflour) — always within the ≤2 ceiling, so it passes, but for a household cooking from what's in stock this could grate. If it does, the fix is one sentence in the isolated `VOICE_AND_WRITING` block (prefer to make the dish sing from what's in stock; reserve to_buy for genuinely transformative additions). Not a problem today; flagged so the fix location is known.
- **Gemini `seed` config field.** `GenerateContentConfig` accepts a `seed` — passing a fixed value would make Gemini's sampling deterministic across eval runs, tightening diffs further than low temperature alone. A CP-polish idea for if/when the voice loop needs even tighter comparison.
- **Voice tuning has no terminal state.** "Complete" here means "hits the examples.md bar and I've stopped finding things to change," not "perfect." Re-run the harness if a future prompt edit risks regressing the tricky briefs (especially the peanut-allergen safety case).
