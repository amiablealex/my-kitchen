## Phase 2 — Catalogue management & data foundations
All four scoped deliverables are built, tested end-to-end, and running on the Pi:

- Alembic introduced and the existing schema baselined without data loss. The live SQLite DB was stamped at the baseline revision (no DDL run, no data touched); fresh machines build the same schema via flask db upgrade. The standing workflow is now git pull → flask db upgrade → restart.
- Ingredient CRUD (name, category, is_staple) under /manage/ingredients, with the existing stock toggle + notes left fully intact.
- Category CRUD (name, section, display_order) — the spec's "configurable categories," pulled forward — with section changes re-laning ingredients in the wizard as intended.
- Equipment CRUD (name, is_available) including a quick availability toggle. Stays un-wired from the generation prompt (deferred to Phase 3, per scope).

Constraints honoured: single-user, no auth, deliberately unstyled skeleton.

## Things learned along the way (not accounted for in scope)

1. Adding a NOT NULL column to a populated table fails on the naive autogenerated migration (Cannot add a NOT NULL column with default value NULL). The fix — add with a temporary server_default, backfill, then drop the default — is now the reusable pattern for every future NOT NULL column. Worth a standing note in the migration workflow.
1. The baseline migration must be autogenerated against an empty DB. Run against the populated DB it produces an empty migration (no diff detected). This is the one non-obvious step in baselining an existing database.
1. One schema addition beyond pure CRUD: Ingredient.is_active, to support retire (hidden from stock list, wizard, and generation brief; kept in history; reactivatable) as the safe default, with hard delete as the explicit secondary action. This was the first real migration applied to live data and doubled as the proof of the upgrade workflow.
1. Category delete needs a guard: ingredients.category_id is NOT NULL, so deleting a non-empty category would orphan rows. We block deletion until it's empty — and the count must include retired ingredients, which still hold a category_id.
1. create_all and Alembic produce textually different but structurally identical DDL (different schema-text hashes, identical columns/constraints). Flagging so a future hash comparison doesn't raise a false alarm.
1. CSRF protection is still absent on the new forms — a deliberate Phase 1 carry-over. It pairs naturally with auth and should be closed in Phase 2 rather than as standalone work.

Forward note for the roadmap
Phases 2+ touch tables that are already in the baseline (dietary_tags, user_dietary_tags, the auth fields on users, favourites flags on recipes), so — like categories and equipment this phase — they should be application-code only, no migration required. The first time a new column is needed, item 1 above applies.
Also still open from the MVP summary and untouched by Phase 1: server-side idempotency / async generation (Phase 5) and local-time display (Phase 4).
